Gargalos Críticos

[C1] Saturação de workers HTTP por streaming DICOM bloqueante CRITICAL

  • Classificação: CRITICAL
  • Local exato: backend/healthlin-backend.service:10-14, backend/proxy.py:10-12, backend/proxy.py:33-55, backend/proxy.py:175-178
  • Descrição objetiva: cada transferência DICOM ocupa worker síncrono por até 120s; com 4 workers, poucas transferências simultâneas podem bloquear a API inteira.
  • Causa técnica: Gunicorn com workers síncronos + proxy com requests.request(..., stream=True) em rota de arquivo pesado.
  • Impacto real: lentidão extrema, timeout e indisponibilidade de rotas críticas (auth/listagens/viewer) sob concorrência moderada.
  • Cenário: múltiplos usuários abrindo estudos grandes ao mesmo tempo.
  • Evidência no código: --workers 4, TIMEOUT_TRANSFER = 120, proxy_request(... stream=True).
  • Tipo de evidência: Fato observado no código local.
  • Recomendação: separar rota de transferência pesada, adotar estratégia de concorrência apropriada (worker class/threading/serviço dedicado) e limites por rota.

[C2] Mudança de usuário pode derrubar o serviço de imagens (restart Orthanc) CRITICAL

  • Classificação: CRITICAL
  • Local exato: backend/app.py:30, backend/auth.py:77, backend/auth.py:158, backend/auth.py:169, backend/auth.py:179, backend/auth.py:204, backend/orthanc_sync.py:23-25, backend/orthanc_sync.py:74-78
  • Descrição objetiva: eventos de login/admin (incluindo troca de senha) disparam sync que pode executar systemctl restart orthanc.
  • Causa técnica: acoplamento direto entre fluxo de autenticação e operação de infraestrutura.
  • Impacto real: interrupção de exames em andamento, erro no viewer e perda de fluidez para todos os usuários conectados.
  • Cenário: criação/desativação/reativação de usuário durante uso ativo do viewer.
  • Evidência no código: user_changed conectado ao request_sync() + subprocess.run(["systemctl", "restart", "orthanc"]).
  • Tipo de evidência: Fato observado no código local.
  • Recomendação: desacoplar para fila assíncrona e aplicar atualização de credenciais sem restart por operação individual.

Problemas de Latência

[H1] Proxy sem pool de conexões HTTP com Orthanc HIGH

  • Classificação: HIGH
  • Local exato: backend/proxy.py:47, backend/proxy.py:216
  • Descrição objetiva: uso de requests.request direto em cada chamada impede reaproveitamento consistente de conexões.
  • Causa técnica: ausência de requests.Session compartilhada com pool/adapter configurado.
  • Impacto real: overhead de handshake/TCP, maior latência por request e menor throughput em bursts de metadados/imagens.
  • Cenário: navegação rápida entre séries/instâncias.
  • Evidência no código: chamadas diretas a requests.request sem session persistente.
  • Tipo de evidência: Fato observado no código local.
  • Recomendação: usar session global com pool, timeout granular e política de retry para erros transitórios idempotentes.

[H2] Busca do painel admin gera tempestade de requests HIGH

  • Classificação: HIGH
  • Local exato: viewer/src/pages/admin/AdminUsersPage.tsx:61-77, viewer/src/pages/admin/AdminUsersPage.tsx:80-83, backend/models.py:130-149
  • Descrição objetiva: cada tecla atualiza search e dispara nova chamada de listagem; não há debounce nem cancelamento.
  • Causa técnica: efeito dependente de estado de filtro sem coordenação assíncrona.
  • Impacto real: maior latência percebida, consumo de CPU no backend e instabilidade visual por respostas fora de ordem.
  • Cenário: digitação contínua com base de usuários crescente.
  • Evidência no código: useEffect(() => { fetchUsers() }, [fetchUsers]) + handleSearchChange por tecla.
  • Tipo de evidência: Fato observado no código local.
  • Recomendação: debounce (ex.: 250-400ms), cancelamento de requests anteriores e guarda por versão de resposta.

[H3] Bundle do viewer muito pesado e com alerta de chunk circular HIGH

  • Classificação: HIGH
  • Local exato: viewer/dist/assets/cornerstone-init-gDCy8D9A.js, viewer/dist/assets/index-l3Ygh61Z.js, viewer/vite.config.ts
  • Descrição objetiva: o chunk principal do runtime de imagem possui 4,036,245 bytes; o build local também reportou warning de dependência circular com risco de ordem de execução quebrada.
  • Causa técnica: concentração de runtime Cornerstone em chunk grande e acoplamento de módulos com reexports circulares.
  • Impacto real: abertura inicial lenta do viewer, maior consumo de memória e risco de falha em runtime em cenários de chunking adversos.
  • Cenário: primeiro acesso ao viewer em máquina/rede fraca.
  • Evidência no código: medição de viewer/dist/assets + saída de npm run -s build com aviso "likely lead to broken execution order".
  • Tipo de evidência: Fato observado localmente.
  • Recomendação: revisar chunking do viewer, reduzir custo do caminho crítico e mitigar o warning circular via estratégia de bundle adequada.

Recomendações de Melhoria

Prioridade 0 (imediata)

  1. Desacoplar restart do Orthanc do fluxo de usuário e remover indisponibilidade induzida por operação administrativa.
  2. Mitigar gargalo de concorrência no proxy de DICOM (arquitetura de workers/streaming).
  3. Adicionar debounce/cancelamento e tratamento robusto de erro no painel admin.

Prioridade 1 (alta)

  1. Aplicar pool de conexões para Orthanc no backend.
  2. Reduzir bundle crítico do viewer e tratar warning circular de build.
  3. Definir tuning explícito de SQLite para concorrência.
  4. Corrigir evento global de offline para não gerar falso-positivo.

Prioridade 2 (sustentação)

  1. Aplicar cache com limite para metadados e cache de séries já visitadas.
  2. Ajustar seletor/memoização em hooks do dashboard para reduzir render desnecessário.
  3. Alinhar contratos de tipos DICOM e padronizar nomenclatura interna.
  4. Introduzir suíte mínima de testes de regressão para fluxos críticos.

Validação Final

Escopo respeitado: nenhuma conclusão depende de servidor remoto, SSH, SCP ou comparação com produção.
