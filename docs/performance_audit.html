Gargalos Críticos

[C1] Backend não inicializa localmente com configuração padrão CRITICAL

  • Local exato: backend/config.py:22, backend/models.py:37, backend/app.py:25.
  • Descrição objetiva: o caminho padrão do banco é absoluto Linux (/opt/healthlin/backend/healthlin.db) e não existe na máquina auditada.
  • Causa técnica: fallback de DB_PATH sem estratégia portátil para ambiente local.
  • Impacto real: indisponibilidade total local (startup falha).
  • Cenário: execução local com python backend/app.py.
  • Evidência no código: falha reproduzida localmente com sqlite3.OperationalError: unable to open database file.
  • Recomendação (sem aplicar correção): fallback local relativo ao repositório e validação explícita de path no boot.

[C2] Operações de auth/usuário disparam restart do Orthanc CRITICAL

  • Local exato: backend/auth.py:53-56, backend/auth.py:82, backend/auth.py:144-145, backend/auth.py:155-156, backend/auth.py:163-164, backend/auth.py:188-189, backend/orthanc_sync.py:59-67.
  • Descrição objetiva: eventos de login/backfill e gestão de usuários acionam sync que reescreve credenciais e reinicia o serviço de imagens.
  • Causa técnica: acoplamento direto de fluxo de usuário com operação pesada de infraestrutura.
  • Impacto real: indisponibilidade temporária do Orthanc e erros 502/504 no viewer.
  • Cenário: ações administrativas durante uso do visualizador.
  • Evidência no código: subprocess.run(["systemctl", "restart", "orthanc"], timeout=30).
  • Recomendação: desacoplar via job serial assíncrono e evitar restart por operação unitária.

Problemas de Latência

[H1] Stack bloqueante com servidor embutido e proxy síncrono HIGH

  • Local exato: backend/app.py:77, backend/healthlin-backend.service:10, backend/proxy.py:49-57.
  • Descrição objetiva: serviço sobe via python app.py e processa chamadas HTTP síncronas para Orthanc com timeout de até 120s.
  • Causa técnica: ausência de servidor de produção com estratégia de workers + I/O bloqueante no request path.
  • Impacto real: fila de requisições, aumento de latência e sensação de travamento sob concorrência.
  • Cenário: múltiplos downloads DICOM simultâneos.
  • Evidência no código: requests.request(..., timeout=timeout, stream=True) em rota HTTP.
  • Recomendação: usar servidor WSGI/ASGI de produção com workers/limites.

[H2] Tratamento incompleto de exceções no proxy gateway HIGH

  • Local exato: backend/proxy.py:92-95.
  • Descrição objetiva: somente ConnectionError e Timeout são tratados; outras exceções de request podem gerar 500.
  • Causa técnica: ausência de fallback para requests.RequestException genérico.
  • Impacto real: falhas sem resposta consistente e pior experiência de erro no frontend.
  • Cenário: erros de protocolo/upstream fora dos dois casos mapeados.
  • Evidência no código: bloco except limitado no final de proxy_request.
  • Recomendação: padronizar tratamento de exceções de gateway.

[H3] Dashboard carrega pacientes em massa sem paginação HIGH

  • Local exato: viewer/src/services/orthanc/patients/index.ts:6, viewer/src/hooks/orthanc/usePatients.ts:12-18, viewer/src/components/dashboard/patient-list/PatientList.tsx:65-90.
  • Descrição objetiva: consulta usa /patients?expand e renderiza lista completa client-side sem virtualização.
  • Causa técnica: desenho de carga total (bulk) sem estratégia incremental.
  • Impacto real: piora de tempo de carregamento e fluidez com crescimento de dados.
  • Cenário: base com centenas/milhares de pacientes.
  • Evidência no código: endpoint expand + tabela mapeada integralmente.
  • Recomendação: paginação server-side, lazy load e virtualização.

[H4] Respostas fora de ordem em estudos por paciente HIGH

  • Local exato: viewer/src/hooks/orthanc/useStudies.ts:8-12, viewer/src/store/dashboard/index.ts:59-66.
  • Descrição objetiva: trocas rápidas de paciente disparam fetches sem cancelamento/versionamento.
  • Causa técnica: ausência de controle de concorrência no fluxo fetchStudies.
  • Impacto real: exames mostrados para paciente errado (inconsistência de interface).
  • Cenário: cliques rápidos com latência variável entre requests.
  • Evidência no código: não há AbortController nem token de versão nesse fluxo.
  • Recomendação: cancelar request anterior e validar versão antes de gravar estado.

Problemas no Frontend

[H5] Viewer com custo alto de carregamento e decode no cliente HIGH

  • Local exato: viewer/vite.config.ts:13-16, viewer/dist/assets/cornerstone-init-D8PD8TbF.js.
  • Descrição objetiva: alias usa bundle NoWebWorkers; chunk de runtime DICOM local mede ~3.85 MiB (3941.6 KiB bruto).
  • Causa técnica: dependência pesada no caminho crítico do viewer e offload de decode limitado.
  • Impacto real: maior tempo de entrada no viewer e risco de jank em máquinas fracas.
  • Cenário: abertura inicial de estudo em hardware modesto.
  • Evidência no código: alias explícito para cornerstoneDICOMImageLoaderNoWebWorkers + medição local do dist.
  • Recomendação: reavaliar pipeline com workers e reduzir peso do bundle crítico.

[H6] Falhas silenciosas em ações administrativas de usuários HIGH

  • Local exato: viewer/src/pages/admin/AdminUsersPage.tsx:41-48, viewer/src/pages/admin/AdminUsersPage.tsx:66-70.
  • Descrição objetiva: listagem e reativação podem falhar sem mensagem consistente ao usuário.
  • Causa técnica: ausência de catch em pontos críticos e feedback incompleto.
  • Impacto real: operador perde confiança no estado exibido e repete ações desnecessárias.
  • Cenário: timeout/falha backend durante administração.
  • Evidência no código: fetchUsers sem branch de erro e handleReactivate sem tratamento.
  • Recomendação: padronizar UX de erro com status por ação.

[M2] Cache de metadados sem limite de memória MEDIUM

  • Local exato: viewer/src/components/viewer/metadata-panel/MetadataPanel.tsx:27, viewer/src/components/viewer/metadata-panel/MetadataPanel.tsx:43.
  • Descrição objetiva: Map cresce por instância visitada sem expiração/evicção.
  • Causa técnica: cache in-memory sem política de tamanho.
  • Impacto real: crescimento gradual de memória em sessões longas.
  • Cenário: navegação extensa por muitos exames.
  • Evidência no código: apenas inserção no cache, sem remoção.
  • Recomendação: adotar LRU/TTL e limite de itens.

[M3] Rerenders e filtros O(n) em assinaturas amplas de store MEDIUM

  • Local exato: viewer/src/hooks/orthanc/usePatients.ts:6-18, viewer/src/hooks/orthanc/useStudies.ts:6-31, viewer/src/components/dashboard/search-bar/SearchBar.tsx:5.
  • Descrição objetiva: dados derivados pesados são recalculados com frequência por dependência de estado amplo.
  • Causa técnica: falta de seletores mínimos e memoização de filtros.
  • Impacto real: perda de fluidez com volume alto de dados.
  • Cenário: dashboard com grande número de pacientes/estudos.
  • Evidência no código: filtros lineares executados no render dos hooks.
  • Recomendação: seletor fino + memoização de coleções filtradas.

Problemas no Backend Python

[H7] Endpoint de usuários sem paginação e sem filtros HIGH

  • Local exato: backend/auth.py:116-120, backend/models.py:105-109.
  • Descrição objetiva: listagem retorna todos os usuários em payload único.
  • Causa técnica: query full table sem LIMIT/OFFSET.
  • Impacto real: latência e custo de serialização crescem com o volume.
  • Cenário: expansão da base de usuários.
  • Evidência no código: SELECT ... FROM users sem paginação.
  • Recomendação: paginação, ordenação e filtros no endpoint.

[M4] Startup inclui tarefas síncronas antes do serviço ficar pronto MEDIUM

  • Local exato: backend/app.py:75-77.
  • Descrição objetiva: criação de admin inicial e sync Orthanc rodam no caminho crítico de boot.
  • Causa técnica: responsabilidades operacionais no startup do processo web.
  • Impacto real: readiness mais lenta e maior janela de indisponibilidade no reinício.
  • Cenário: restart com Orthanc lento ou indisponível.
  • Evidência no código: chamadas síncronas imediatamente antes de app.run.
  • Recomendação: mover para job pós-inicialização com monitoramento.

Problemas no SQLite

[H8] Configuração SQLite mínima para concorrência real HIGH

  • Local exato: backend/models.py:36-39.
  • Descrição objetiva: conexão criada sem busy_timeout, sem WAL e sem política de contenção.
  • Causa técnica: uso da configuração padrão do driver.
  • Impacto real: risco de lock em escrita concorrente e falhas intermitentes.
  • Cenário: requests simultâneos de auth/admin.
  • Evidência no código: apenas sqlite3.connect(config.DB_PATH).
  • Recomendação: configurar timeout/journal e plano de migração para banco mais robusto ao escalar.

[M1] Migração executada a cada boot com captura genérica de erro MEDIUM

  • Local exato: backend/models.py:59-62.
  • Descrição objetiva: ALTER TABLE sempre executado e qualquer erro é suprimido.
  • Causa técnica: ausência de versionamento de schema e except Exception: pass.
  • Impacto real: falhas reais de banco podem passar despercebidas.
  • Cenário: lock/corrupção/permissão no arquivo DB durante inicialização.
  • Evidência no código: captura ampla no bloco de migração.
  • Recomendação: controlar versão de schema e tratar apenas erro esperado de coluna existente.

Problemas na Integração Frontend/Backend

[H9] UI depende de carregamento monolítico para abrir dashboard HIGH

  • Local exato: viewer/src/hooks/orthanc/usePatients.ts:8-10, viewer/src/store/dashboard/index.ts:42-56.
  • Descrição objetiva: sem payload inicial de pacientes a tela principal não progride para conteúdo útil.
  • Causa técnica: falta de pipeline incremental e cache para primeira renderização.
  • Impacto real: atraso perceptível na abertura do sistema e UX frágil em rede lenta.
  • Cenário: backend lento / Orthanc com volume alto.
  • Evidência no código: fetch obrigatório no mount para preencher lista base.
  • Recomendação: adotar carregamento progressivo e cache de navegação.

[M5] Cache global de credenciais sem limite e sem lock dedicado MEDIUM

  • Local exato: backend/models.py:24-34, backend/models.py:153-175.
  • Descrição objetiva: cache de credenciais cresce por usuário autenticado, sem eviction por tamanho e sem sincronização explícita.
  • Causa técnica: estrutura global simples com TTL temporal.
  • Impacto real: aumento potencial de memória e comportamento menos previsível sob alta concorrência.
  • Cenário: muitas contas com uso intercalado na mesma janela.
  • Evidência no código: dicionário global sem política de capacidade.
  • Recomendação: cache thread-safe com limite de itens e métricas.

Problemas de Estabilidade

[M6] Interceptor 401 força reload completo e perda de estado MEDIUM

  • Local exato: viewer/src/services/api.ts:24-28.
  • Descrição objetiva: qualquer 401 remove token e força navegação hard para login.
  • Causa técnica: tratamento global agressivo sem distinção de fluxo.
  • Impacto real: interrupção brusca da sessão e perda de contexto visual.
  • Cenário: expiração de token com múltiplas requisições em paralelo.
  • Evidência no código: uso de window.location.href no interceptor global.
  • Recomendação: fluxo SPA controlado e tratamento de expiração mais granular.

[M7] Reativação de usuário pode retornar sucesso sem alterar dados MEDIUM

  • Local exato: backend/models.py:143-147, backend/auth.py:159-164.
  • Descrição objetiva: endpoint responde \"Usuário reativado\" sem verificar se o ID existe.
  • Causa técnica: update sem validação de linhas afetadas.
  • Impacto real: inconsistência funcional e diagnóstico incorreto para o usuário administrativo.
  • Cenário: tentativa de reativar ID inválido.
  • Evidência no código: retorno fixo no endpoint após chamada do model.
  • Recomendação: validar resultado no banco e retornar 404 quando aplicável.

Problemas de Arquitetura

[H10] Acoplamento alto entre autenticação e infraestrutura Orthanc HIGH

  • Local exato: backend/auth.py, backend/orthanc_sync.py.
  • Descrição objetiva: rotas de auth/usuário controlam diretamente sincronização e restart de serviço externo.
  • Causa técnica: mistura de responsabilidades entre camada de negócio e operação de infraestrutura.
  • Impacto real: manutenção difícil, efeito cascata de falha e baixa escalabilidade operacional.
  • Cenário: crescimento de operações administrativas ou mudança na política de autenticação.
  • Evidência no código: chamadas recorrentes a orthanc_sync.request_sync() em múltiplas rotas.
  • Recomendação: separar em serviço assíncrono/event-driven.

[L1] Inconsistência de nomenclatura (Ortahnc) reduz manutenibilidade LOW

  • Local exato: viewer/src/store/dashboard/index.ts:20, viewer/src/store/viewer/index.ts:39 e usos derivados.
  • Descrição objetiva: estado offline possui typo recorrente.
  • Causa técnica: padronização de naming incompleta.
  • Impacto real: dificuldade de manutenção e maior chance de erro em refatorações.
  • Cenário: onboarding e evolução do código.
  • Evidência no código: ocorrências repetidas de isOrtahncOffline.
  • Recomendação: renomeação gradual e padronizada.

[L2] Healthcheck superficial não valida dependências críticas LOW

  • Local exato: backend/app.py:27-29.
  • Descrição objetiva: endpoint /api/health retorna {"status":"ok"} sem testar DB/Orthanc.
  • Causa técnica: healthcheck sem profundidade operacional.
  • Impacto real: falso positivo de disponibilidade durante falhas parciais.
  • Cenário: backend ativo, mas dependências indisponíveis.
  • Evidência no código: retorno estático sem verificação de dependências.
  • Recomendação: healthchecks por componente com status detalhado.

Riscos de Escalabilidade

Cenários de Falha (Simulação Lógica com Base no Código Local)

  • Internet/rede lenta: abertura do viewer piora por chunk pesado e decode no cliente (H5).
  • API indisponível: dashboard depende de carga monolítica inicial; admin pode falhar sem feedback robusto (H6, H9).
  • Falha parcial de resposta: proxy sem tratamento genérico pode retornar 500 não padronizado (H2).
  • Banco bloqueado: ausência de busy_timeout aumenta risco de lock sob concorrência (H8).
  • Dados incompletos: reativação pode retornar sucesso falso (M7).
  • Alto volume de registros: degradação forte por ausência de paginação/virtualização (H3, H7).
  • Requests simultâneos: stack síncrona com servidor embutido gera filas e lentidão (H1).
  • Frontend antes do backend pronto: bootstrap de auth pode redirecionar e perder contexto de sessão (M6).
  • Erro de autenticação: 401 global aciona reload completo da SPA (M6).
  • Timeout/exceção inesperada: apenas subset de exceções é tratado no gateway (H2).

Recomendações de Melhoria

Prioridade imediata

  1. Tornar DB_PATH portátil para execução local e validar dependências no startup.
  2. Remover restart de Orthanc do fluxo de auth; usar pipeline assíncrono serializado.
  3. Migrar execução para servidor de produção com workers e limites por rota.
  4. Implementar paginação/carregamento incremental em pacientes e usuários.

Prioridade alta

  1. Adicionar cancelamento/versionamento em fetchStudies.
  2. Revisar pipeline do viewer para web workers e menor custo de bundle crítico.
  3. Padronizar tratamento de erro no proxy e nos fluxos administrativos.
  4. Configurar SQLite para concorrência segura e planejar evolução de persistência.

Prioridade contínua

  1. Aplicar políticas de eviction em caches de frontend/backend.
  2. Reduzir rerenders com seletores finos e memoização.
  3. Adicionar testes automatizados para auth/proxy/dashboard/viewer.
