Gargalos Críticos

[C1] Sessão JWT sem revalidação de usuário + fallback para credencial global do Orthanc CRITICAL

  • Classificação: CRITICAL
  • Local exato: backend/auth.py (require_auth, require_admin), backend/proxy.py (_get_orthanc_auth), backend/models.py (get_user_credentials)
  • Status de verificação: Fato observado diretamente no código local.
  • Descrição objetiva: as rotas protegidas validam só o token JWT; se a credencial do usuário não existir/estiver inativa, o proxy usa fallback para ORTHANC_USER/ORTHANC_PASS.
  • Causa técnica: ausência de validação de active por request + política de fallback permissiva no proxy.
  • Impacto real no usuário/sistema: inconsistência de autorização (conta desativada pode continuar operando com token válido) e risco de acesso com credencial global.
  • Cenário em que aparece: usuário é desativado após login; token ainda não expirou.
  • Evidência no código: require_auth não consulta DB; get_user_credentials(... active = 1) pode retornar None; em seguida return (config.ORTHANC_USER, config.ORTHANC_PASS).
  • Recomendação (sem aplicar correção): revalidar usuário ativo nas rotas autenticadas e negar acesso quando faltar credencial individual; limitar fallback global a fluxo explícito de serviço técnico.

[C2] Operações de usuário acionam sync com restart do Orthanc CRITICAL

  • Classificação: CRITICAL
  • Local exato: backend/auth.py, backend/app.py, backend/orthanc_sync.py
  • Status de verificação: Fato observado diretamente no código local.
  • Descrição objetiva: login (backfill), criação, desativação, reativação e troca de senha disparam user_changed, que agenda sync e pode executar systemctl restart orthanc.
  • Causa técnica: acoplamento direto entre fluxo de autenticação/usuário e operação de infraestrutura.
  • Impacto real no usuário/sistema: janelas de indisponibilidade de imagens (502/504), perda de fluidez e risco de interrupção do fluxo.
  • Cenário em que aparece: administração de usuários durante uso ativo do viewer.
  • Evidência no código: subprocess.run(["systemctl", "restart", "orthanc"], timeout=30) em sync_orthanc_users().
  • Recomendação (sem aplicar correção): desacoplar por fila assíncrona, consolidar alterações e aplicar atualização sem restart por operação unitária.

Problemas de Latência

[H1] Gargalo de concorrência: proxy síncrono + workers limitados HIGH

  • Classificação: HIGH
  • Local exato: backend/healthlin-backend.service, backend/proxy.py
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: serviço usa Gunicorn padrão síncrono (--workers 4) e o proxy executa chamadas bloqueantes para o Orthanc, incluindo transferências longas.
  • Causa técnica: cada request de download/processamento ocupa um worker inteiro até o término.
  • Impacto real no usuário/sistema: aumento de fila, respostas lentas e travamentos perceptíveis sob múltiplos acessos simultâneos.
  • Cenário em que aparece: vários downloads de DICOM e consultas em paralelo.
  • Evidência no código: requests.request(... stream=True, timeout=120) + --workers 4.
  • Recomendação: separar tráfego pesado, adotar estratégia de concorrência adequada e limite de transferência por rota.

[H3] Entrada do viewer com asset muito pesado e alias sem web workers HIGH

  • Classificação: HIGH
  • Local exato: viewer/vite.config.ts, viewer/dist/assets/cornerstone-init-CXRo3yXp.js
  • Status de verificação: Fato observado no código/build local.
  • Descrição objetiva: a rota do viewer carrega chunk de ~3.85 MiB bruto e o alias força bundle NoWebWorkers do dicom-image-loader.
  • Causa técnica: runtime de imagem concentrado em chunk grande + menor offload de decode para worker.
  • Impacto real no usuário/sistema: tempo maior para abrir viewer, maior risco de jank em máquinas fracas e rede lenta.
  • Cenário em que aparece: primeiro acesso ao viewer após login.
  • Evidência no código: alias para cornerstoneDICOMImageLoaderNoWebWorkers.bundle.min.js; asset local medido em 4,036,245 bytes.
  • Recomendação: revisar estratégia de chunking e decode, priorizando worker real e redução do custo no caminho crítico.

[H4] Busca de usuários sem debounce/cancelamento gera tempestade de requests HIGH

  • Classificação: HIGH
  • Local exato: viewer/src/pages/admin/AdminUsersPage.tsx, backend/models.py (list_users)
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: cada tecla muda search e dispara nova listagem; não há debounce, abort de request anterior, nem controle de resposta fora de ordem.
  • Causa técnica: useEffect(() => fetchUsers(), [fetchUsers]) com dependência de search e sem coordenação assíncrona.
  • Impacto real no usuário/sistema: latência percebida, consumo extra de CPU/DB e possível oscilação visual de resultados.
  • Cenário em que aparece: digitação rápida com base de usuários maior.
  • Evidência no código: handleSearchChange atualiza estado a cada tecla; backend executa COUNT(*) + SELECT ... LIKE '%...%' por requisição.
  • Recomendação: implementar debounce, cancelamento por AbortController e versionamento da resposta.

Problemas no Frontend

[M1] Re-renders evitáveis por assinatura ampla do Zustand e filtros O(n) no render MEDIUM

  • Classificação: MEDIUM
  • Local exato: viewer/src/hooks/orthanc/usePatients.ts, viewer/src/hooks/orthanc/useStudies.ts, viewer/src/components/dashboard/search-bar/SearchBar.tsx
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: hooks usam useDashboardStore() sem seletor fino e filtram listas em cada render.
  • Causa técnica: subscription ampla + ausência de memoização de derivação.
  • Impacto real no usuário/sistema: degradação gradual de fluidez quando cresce volume de pacientes/estudos.
  • Cenário em que aparece: dashboard com navegação intensa e filtros ativos.
  • Evidência no código: uso direto de patients.filter(...) e studies.filter(...) dentro dos hooks.
  • Recomendação: usar seletores mínimos e memoização dos filtros derivados.

[M2] Cache de metadados sem política de limite/evicção MEDIUM

  • Classificação: MEDIUM
  • Local exato: viewer/src/components/viewer/metadata-panel/MetadataPanel.tsx
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: Map<instanceId, tags> cresce durante a sessão e nunca remove entradas.
  • Causa técnica: cache em memória sem TTL/LRU/tamanho máximo.
  • Impacto real no usuário/sistema: consumo crescente de memória em sessões longas de leitura.
  • Cenário em que aparece: navegação por grande número de instâncias/exames.
  • Evidência no código: apenas cacheRef.current.set(...), sem rotina de limpeza.
  • Recomendação: aplicar política de evicção (LRU/TTL) com limite explícito.

[M3] Navegação de frames sem prefetch e com recarga de stack por mudança MEDIUM

  • Classificação: MEDIUM
  • Local exato: viewer/src/components/viewer/image-panel/ImagePanel.tsx, viewer/src/store/viewer/index.ts, viewer/src/components/viewer/dicom-canvas/DicomCanvas.tsx
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: cada mudança de frame atualiza currentInstance e aciona novo setStack no viewport.
  • Causa técnica: pipeline orientado a troca de instância individual, sem prefetch local de vizinhos.
  • Impacto real no usuário/sistema: sensação de atraso no scroll/slider em séries grandes.
  • Cenário em que aparece: arraste rápido do slider em estudo com muitas imagens.
  • Evidência no código: setFrame -> currentInstance e await viewport.setStack(imageIds, 0) por atualização.
  • Recomendação: adotar prefetch/cache de imagens adjacentes e reduzir recarga completa da stack.

[L2] Código morto em hooks não utilizados aumenta custo de manutenção LOW

  • Cenário em que aparece: onboarding e refatorações de viewer/auth.

Problemas no Backend Python

[H2] Inicialização com efeitos colaterais duplicados por worker HIGH

  • Classificação: HIGH
  • Local exato: backend/app.py (create_app), backend/healthlin-backend.service, backend/orthanc_sync.py
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: cada worker executa init_db(), create_initial_admin() e thread de sync inicial.
  • Causa técnica: rotina de bootstrap com tarefas operacionais dentro do ciclo de criação da aplicação web por processo.
  • Impacto real no usuário/sistema: boot mais lento, risco de corrida na inicialização e variação de disponibilidade após restart.
  • Cenário em que aparece: subida do serviço com --workers 4.
  • Evidência no código: thread orthanc-init-sync criada em create_app(); locks de sync são apenas threading.Lock (não interprocesso).
  • Recomendação: mover tarefas de migração/sync para job único de bootstrap fora dos workers HTTP.

[M4] Tratamento de erro incompleto na administração de usuários MEDIUM

  • Classificação: MEDIUM
  • Local exato: viewer/src/pages/admin/AdminUsersPage.tsx, rotas /api/auth/users em backend/auth.py
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: algumas falhas da camada API não geram feedback consistente ao operador.
  • Causa técnica: ausência de catch em partes do fluxo de listagem/reativação.
  • Impacto real no usuário/sistema: ações parecem “não responder”, com estado possivelmente desatualizado na tela.
  • Cenário em que aparece: timeout/erro transitório durante gestão de usuários.
  • Evidência no código: fetchUsers sem bloco de erro explícito para UI; handleReactivate sem tratamento de exceção.
  • Recomendação: padronizar mensagens de falha e estados de erro por ação.

[M6] Healthcheck superficial não valida dependências críticas MEDIUM

  • Classificação: MEDIUM
  • Local exato: backend/app.py (/api/health)
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: endpoint retorna sempre {"status":"ok"} sem validar SQLite/Orthanc.
  • Causa técnica: ausência de checks de dependência no health endpoint.
  • Impacto real no usuário/sistema: falso positivo de disponibilidade em falhas parciais.
  • Cenário em que aparece: backend no ar, mas Orthanc indisponível.
  • Evidência no código: retorno estático no handler de saúde.
  • Recomendação: healthcheck composto (DB + proxy Orthanc + estado interno de sync) com status detalhado.

Problemas no SQLite

[H5] Consulta de usuários com padrão que degrada com crescimento de dados HIGH

  • Classificação: HIGH
  • Local exato: backend/models.py (list_users), schema em init_db()
  • Status de verificação: Fato observado no código local.
  • Descrição objetiva: para cada listagem, executa COUNT(*) + SELECT ... ORDER BY name, com busca LIKE '%termo%'.
  • Causa técnica: custo de varredura e ordenação aumenta com o volume; schema não declara índices para role/active/name.
  • Impacto real no usuário/sistema: latência crescente no painel admin e maior uso de CPU com base maior.
  • Cenário em que aparece: organização com muitos usuários e uso frequente de busca/filtro.
  • Evidência no código: query dinâmica em list_users e schema com apenas PRIMARY KEY + UNIQUE(council_type, council_number).
  • Recomendação: revisar plano de indexação e estratégia de busca (ex.: índice apropriado ou FTS para texto).

[M5] Conexão SQLite sem tuning explícito para concorrência/locks MEDIUM

  • Classificação: MEDIUM
  • Local exato: backend/models.py (get_db)
  • Status de verificação: Fato no código; impacto quantitativo depende do volume (não verificável sem DB real nesta cópia local).
  • Descrição objetiva: conexão criada com defaults e sem PRAGMAs explícitos de concorrência.
  • Causa técnica: ausência de configuração específica de journal/timeout/padrão de escrita para a carga esperada.
  • Impacto real no usuário/sistema: maior risco de lock e latência em cenários de escrita concorrente.
  • Cenário em que aparece: múltiplas operações administrativas simultâneas.
  • Evidência no código: apenas sqlite3.connect(config.DB_PATH); não há PRAGMA no módulo.
  • Recomendação: definir política explícita de concorrência e plano de evolução do banco quando o volume crescer.

Recomendações de Melhoria

Prioridade imediata (risco crítico/alto)

  1. Bloquear fallback global de credencial para sessões de usuário e revalidar usuário ativo em toda rota autenticada.
  2. Remover restart do Orthanc do caminho de operações de usuário; usar fila assíncrona com consolidação.
  3. Mitigar gargalo do proxy (concorrência/workers/rota de transferência) para evitar saturação com múltiplos viewers.
  4. Debounce + cancelamento na busca de usuários e proteção contra respostas fora de ordem.

Prioridade alta (estabilidade e escala)

  1. Revisar estratégia de build do viewer (peso de chunk e uso de workers de decode).
  2. Revisar queries de listagem de usuários (busca, ordenação e indexação condizente com crescimento).
  3. Criar healthcheck composto para DB + Orthanc + estado interno de sync.
  4. Definir política explícita de concorrência SQLite e plano de evolução para maior escala.

Prioridade contínua (manutenção)

  1. Aplicar limites de cache em memória no frontend.
  2. Reduzir assinaturas amplas de store e custo de filtros no render.
  3. Eliminar código morto e inconsistências de nomenclatura para reduzir dívida técnica.

Validação Final

Fatos observados: todos os achados classificados como “fato” foram derivados de inspeção direta do código local e dos artefatos locais de build.

Hipóteses explicitadas: cenários marcados como hipótese dependem de resposta runtime do Orthanc e/ou volume real de dados, inexistentes nesta cópia local.

Escopo respeitado: nenhuma conclusão depende de acesso ao servidor remoto; nenhuma ação foi executada em ambiente externo.
