1) VISAO GERAL DO SISTEMA (ARQUITETURA)
====================================================================

Topologia logica:
1. Browser (SPA React/Vite) - UI de login, dashboard, admin e viewer DICOM.
2. Backend Flask - API de autenticacao, autorizacao, proxy autenticado para Orthanc e servidor de arquivos do frontend buildado.
3. SQLite - persistencia de usuarios locais da aplicacao (identidade, role, hash e senha criptografada).
4. Orthanc - repositorio PACS/DICOM externo consumido via proxy backend.
5. Systemd/OS - processo do backend, variaveis de ambiente e reinicio do Orthanc no sync de credenciais.

Composicao por modulo:
- Entrypoint backend: backend/app.py
  - Cria app Flask com CORS em /api/*.
  - Registra blueprints auth e proxy.
  - Inicializa DB (models.init_db).
  - Exponibiliza /api/health.
  - Faz SPA fallback para FRONTEND_DIST.
  - Em runtime principal: cria admin inicial e roda sync de usuarios no Orthanc.

- Autenticacao/autorizacao: backend/auth.py
  - JWT HS256 com exp de 12h.
  - Decorators require_auth e require_admin.
  - Rotas de login, me, troca de senha e gestao de usuarios (admin).

- Persistencia: backend/models.py
  - SQLite com tabela users.
  - Hash de senha com bcrypt.
  - password_encrypted com Fernet derivada de JWT_SECRET.
  - Regras de desativacao (admin principal, auto-desativacao, lockout de ultimo admin).

- Proxy Orthanc: backend/proxy.py
  - Rotas /api/orthanc/* autenticadas por JWT.
  - Repassa para ORTHANC_URL via requests + Basic Auth.
  - Credencial preferencial: do usuario logado (council_number + senha descriptografada).
  - Fallback: ORTHANC_USER/ORTHANC_PASS globais.

- Sincronizacao Orthanc: backend/orthanc_sync.py
  - Gera/reescreve arquivo credentials.json do Orthanc.
  - Insere todos usuarios ativos + admin global.
  - Reinicia servico orthanc via systemctl.

- Frontend React: viewer/src
  - Router com rotas lazy (dashboard, viewer, admin users).
  - Zustand para 3 dominos de estado: auth, dashboard, viewer.
  - Axios para API geral + interceptors (JWT/erros).
  - Cornerstone3D para renderizacao DICOM com loader wadouri.

Padroes observados:
- Backend usa separacao por modulos (config, models, auth blueprint, proxy blueprint), sem ORM.
- Frontend usa separacao por tipo de modulo (pages/components/hooks/services/store/types).
- Estado global distribuido em stores Zustand por dominio.
- Integracao HTTP centralizada em services.

Ausencias arquiteturais relevantes:
- Sem camada formal de servico no backend (regra de negocio distribuida entre rota e models).
- Sem framework de migracao de banco (alter table manual no init_db).
- Sem suite de testes implementada (pastas de teste existem, mas sem arquivos).
- Sem esquema de validacao de payload (ex.: pydantic/zod) no backend/frontend.

2) DIAGRAMA LOGICO TEXTUAL (FLUXO END-TO-END)
====================================================================

Fluxo macro:
[Browser React]
  -> POST /api/auth/login
[Flask auth blueprint]
  -> models.authenticate (SQLite + bcrypt)
  -> JWT (HS256)
  <- token + user
[Browser]
  -> localStorage.healthlin_token
  -> GET /api/auth/me (hidrata sessao)

[Dashboard]
  -> GET /api/orthanc/patients
[Flask proxy]
  -> requests -> ORTHANC_URL/patients (Basic Auth via _get_orthanc_auth)
  <- lista IDs
[Browser]
  -> N chamadas GET /api/orthanc/patients/{id}

[Selecao de estudo]
  -> GET /api/orthanc/studies/{studyId}
  -> GET /api/orthanc/series/{seriesId}
  -> (fallback opcional) GET /api/orthanc/series/{seriesId}/instances

[Viewer DICOM]
  imageId = wadouri:{VITE_API_URL}/api/orthanc/instances/{instanceId}/file
  -> Cornerstone DICOM loader (XHR com Authorization Bearer)
  -> Flask proxy /api/orthanc/instances/{instanceId}/file
  -> Orthanc /instances/{instanceId}/file (Basic Auth)
  <- bytes DICOM
  <- viewport.setStack([imageId],0)
  -> render no canvas

[Metadados]
  -> GET /api/orthanc/instances/{instanceId}/simplified-tags
  -> render painel lateral

Fluxo de credenciais Orthanc:
[CRUD usuario / troca senha]
  -> auth._sync_orthanc() (thread)
  -> orthanc_sync.sync_orthanc_users()
  -> escreve ORTHANC_CREDENTIALS_PATH
  -> systemctl restart orthanc

3.2.3 Autenticacao e permissao

Arquivo: backend/auth.py

Token:
- create_token(user): JWT HS256 com payload
  - user_id, name, role, council_type
  - exp UTC agora + 12h

Decorators:
- require_auth:
  - le Authorization Bearer
  - decode JWT com JWT_SECRET
  - injeta g.user
  - retorna 401 para token ausente/expirado/invalido
- require_admin:
  - require_auth + role == admin
  - retorna 403 caso contrario

Rotas:
- POST /api/auth/login
  - valida payload
  - models.authenticate(council_type, council_number, password)
  - se sucesso:
    - models.update_password_encrypted(user_id, password) (backfill)
    - retorna token + user
  - falhas:
    - 400 dados ausentes/campos
    - 401 credenciais invalidas

- GET /api/auth/me
  - require_auth
  - retorna dados do usuario ativo
  - 404 se usuario nao encontrado

- GET /api/auth/users (admin)
- POST /api/auth/users (admin)
  - valida role em admin|medico|tecnico
  - create_user no models
  - sucesso 201
  - conflito 409
  - dispara _sync_orthanc async

- DELETE /api/auth/users/{id} (admin)
  - models.deactivate_user
  - retorna 403 para violacao de regra de negocio
  - dispara _sync_orthanc async

- PUT /api/auth/users/{id}/reactivate (admin)
  - reativa e dispara _sync_orthanc

- PUT /api/auth/change-password
  - require_auth
  - valida campos e delega a models.change_password
  - sucesso dispara _sync_orthanc

Sync assinc:
- _sync_orthanc() abre thread daemon para sync_orthanc_users.

3.2.4 Modelo de dados (SQLite)

Arquivo: backend/models.py

Tabela users:
- id INTEGER PK AUTOINCREMENT
- name TEXT NOT NULL
- council_type TEXT CHECK in ('CRM','CRTR','MATRICULA')
- council_number TEXT NOT NULL
- password_hash TEXT NOT NULL
- password_encrypted TEXT
- role TEXT CHECK in ('admin','medico','tecnico') default 'medico'
- active INTEGER default 1
- created_at TEXT default datetime('now')
- UNIQUE(council_type, council_number)

Migracao embutida:
- init_db tenta ALTER TABLE users ADD COLUMN password_encrypted.

Senha:
- password_hash: bcrypt
- password_encrypted: Fernet
  - chave derivada deterministicamente de SHA256(JWT_SECRET)

Funcoes relevantes:
- authenticate: verifica active=1 + bcrypt
- get_user_credentials(user_id): devolve (council_number, senha_plain) descriptografada
- list_users_with_credentials(): usuarios ativos com senha plaintext para sync Orthanc
- change_password:
  - valida senha atual
  - exige len(new_password) >= 8
  - atualiza hash + encrypted

Regras de desativacao:
- id=1 nao pode ser desativado
- usuario nao pode desativar a si proprio
- impede desativar ultimo admin ativo (alem do id=1)

3.2.5 Proxy para Orthanc

Arquivo: backend/proxy.py

Blueprint:
- prefixo /api/orthanc
- todas rotas com require_auth

Autenticacao com Orthanc:
- _get_orthanc_auth()
  - tenta credencial do usuario logado via models.get_user_credentials(g.user.user_id)
  - fallback: ORTHANC_USER/ORTHANC_PASS globais

Mecanica de proxy:
- proxy_request(orthanc_path)
  - monta URL final: ORTHANC_URL + path + querystring
  - filtra headers de request (nao repassa host, authorization, content-length, transfer-encoding)
  - envia requests.request(method,url,headers,data,auth,timeout=30,stream=True)
  - filtra headers de resposta (remove content-encoding, transfer-encoding, content-length, www-authenticate)
  - converte 401 do Orthanc para 502 JSON
  - repassa status original para outros casos
  - ConnectionError -> 502
  - Timeout -> 504

3.3 INTEGRACAO ORTHANC (NIVEL BAIXO)
--------------------------------------------------------------------

3.3.1 Como o sistema autentica no Orthanc

Camada 1 (usuario app -> backend):
- Browser envia Bearer JWT ao backend (/api/orthanc/*).
- Backend valida JWT em require_auth e identifica g.user.user_id.

Camada 2 (backend -> Orthanc):
- Proxy chama _get_orthanc_auth().
- Tentativa primaria:
  - models.get_user_credentials(user_id)
  - retorna (council_number, senha_plain descriptografada)
- Fallback:
  - (ORTHANC_USER, ORTHANC_PASS)
- Requests para Orthanc usam Basic Auth nativo do requests(auth=tuple).

3.3.2 Como as requisicoes sao feitas (proxy)

- Path frontend: /api/orthanc/... (sempre no backend Flask)
- Path backend->Orthanc: ORTHANC_URL + orthanc_path
- Query string preservada
- Header Authorization do cliente nao e repassado ao Orthanc
- Conteudo binario DICOM de /instances/{id}/file retorna no corpo da resposta Flask

3.3.3 Fluxo de carregamento DICOM no viewer

1) Viewer seleciona instancia corrente (currentInstance.ID)
2) instancesService.getFileUrl monta imageId:
   - wadouri:{VITE_API_URL}/api/orthanc/instances/{id}/file
3) Cornerstone DICOM loader resolve imageId wadouri
4) beforeSend do loader injeta Bearer JWT no XHR
5) Flask valida JWT (require_auth)
6) Proxy chama Orthanc /instances/{id}/file com Basic Auth
7) bytes retornam ao loader
8) viewport.setStack([imageId],0)
9) viewport.render()

3.3.4 Construcao das URLs wadouri

Implementacao:
- viewer/src/services/orthanc/instances/index.ts
  getFileUrl(id) => `wadouri:${VITE_API_URL ?? ''}/api/orthanc/instances/${id}/file`

Comportamentos observaveis:
- Com VITE_API_URL vazio, bundle atual compila para `wadouri:/api/orthanc/instances/{id}/file`.
- Com VITE_API_URL definido, compila para `wadouri:https://host.../api/orthanc/...`.

3.3.5 Dependencia entre credenciais do app e do Orthanc

Dependencia estrutural central:
- A senha do usuario da aplicacao e a mesma senha usada para conta Basic Auth no Orthanc (via sync de RegisteredUsers).
- create_user / change_password / deactivate / reactivate disparam sync_orthanc.
- Startup do backend tambem dispara sync_orthanc.

Consequencia de acoplamento:
- Estado de autenticacao no Orthanc depende diretamente de consistencia de:
  - DB users (password_encrypted)
  - JWT_SECRET (chave de descriptografia Fernet)
  - escrita em ORTHANC_CREDENTIALS_PATH
  - restart do servico orthanc

5) COMO O ORTHANC E INTEGRADO (NIVEL BAIXO)
====================================================================

A. Contrato de autenticacao
- JWT protege entrada do backend.
- Basic Auth protege backend->Orthanc.
- Mapeamento identidade app->Orthanc:
  - username Orthanc = council_number
  - password Orthanc = senha da aplicacao (descriptografada)

B. Sync de credenciais
- Trigger de sync:
  - startup backend
  - create user
  - deactivate user
  - reactivate user
  - change password
- Efeito:
  - rewrite total de RegisteredUsers
  - restart orthanc

C. Requisicao de imagem
- URL externa de consumo no frontend:
  - /api/orthanc/instances/{id}/file
- URL interna no Orthanc:
  - {ORTHANC_URL}/instances/{id}/file
- Transporte:
  - backend usa requests com stream=True, resposta final via resp.content

D. Endpoints Orthanc proxied
- Leitura de pacientes, estudos, series, instancias
- Busca tools/find e tools/lookup
- Tags simplificadas e completas
- Preview por instancia/frame

E. Integracao DICOMweb
- config-no-local/orthanc-json.txt habilita DicomWeb/EnableWado.
- fluxo do app atual usa endpoints REST de instancia (/instances/{id}/file), nao usa DicomWeb/WadoRoot diretamente.

6) LISTA DE PONTOS FRAGEIS ESTRUTURAIS
====================================================================

1. Chave JWT e chave de criptografia de senha acopladas
- models._get_cipher deriva Fernet de JWT_SECRET.
- Rotacao de JWT_SECRET altera capacidade de descriptografar password_encrypted existente.
- Impacta autenticacao Orthanc baseada em credenciais de usuario.

2. Acoplamento forte identidade app <-> credencial Orthanc
- Senha da aplicacao e propagada para Orthanc.
- Mudancas de usuario dependem de sync + restart do Orthanc.

3. Sync Orthanc desacoplado da resposta HTTP
- auth._sync_orthanc roda em thread daemon, sem bloqueio do request.
- API pode retornar sucesso de CRUD/troca de senha antes de confirmar aplicacao no Orthanc.

4. useCornerstone com singleton global mutavel em modulo
- _initialized/_engine/_toolGroup compartilham estado global entre componentes.
- lifecycle depende de beforeunload para cleanup final.

5. DicomCanvas concentra responsabilidades de infraestrutura + UI
- gerencia viewport lifecycle, resize, carga DICOM, estados offline, overlay de erro/loading.

6. Dependencia do fallback global de credencial Orthanc
- se get_user_credentials falha, proxy usa ORTHANC_USER/ORTHANC_PASS global.
- muda semantica de autenticacao por usuario para credencial compartilhada.

7. Dependencia de privilegio root no processo backend
- service configurado com User=root.
- escrita em /etc/orthanc/credentials.json e restart do Orthanc assumem esse contexto.

8. Dist artefatos duplicados em formatos diferentes
- viewer/dist atual e viewer/healthlin-dist.tgz antigo coexistem com hashes distintos.
- coexistencia adiciona acoplamento operacional no processo de release.

9. Camada de dominio sem fronteira formal
- regras em rotas + models + stores frontend, sem contrato central de dominio.

10. Nomenclatura de estado offline com typo consolidado
- isOrtahncOffline existe em types/stores/hooks/componentes.
- impacta legibilidade transversal do dominio Orthanc.

8) PONTOS ONDE BUGS SAO MAIS PROVAVEIS (SEM PROPOSTA DE CORRECAO)
====================================================================

1. Corridas assinc no ciclo viewer
- loadStudy/selectSeries em paralelo (mitigado por version counters, mas ainda sensivel a troca rapida de rota/serie/frame).

2. Inicializacao/reativacao do viewport
- enableElement/disableElement + resize + remount em DicomCanvas tem varios caminhos de estado.

3. Fronteira axios vs loader DICOM
- REST usa interceptor axios; arquivo DICOM usa XHR do loader.
- duas pilhas de tratamento de erro/autenticacao coexistem.

4. MetadataPanel sem controle de concorrencia
- requests de tags podem retornar fora de ordem em troca rapida de instancia.

5. Sincronizacao de credenciais Orthanc
- depende de IO de arquivo + restart de servico externo.
- falhas podem nao refletir imediatamente na UX da API que acionou o sync.

6. Dependencia de string-path no ecossistema DICOM
- imageId wadouri, route Flask, path proxy e path Orthanc precisam permanecer consistentes.

7. Fallback de auth do proxy
- falha de credencial por usuario redireciona para credencial global.
- comportamento final depende do estado de ORTHANC_USER/ORTHANC_PASS.

8. Configuracao de ambiente divergente entre dev/prod
- frontend pode rodar com baseURL vazio (proxy Vite) e backend em host diferente em producao.
- getFileUrl de DICOM usa composicao separada de axios.

9. Ausencia de testes automatizados no repositrio
- pastas de teste existem, mas sem casos executaveis.
- regressao funcional depende apenas de validacao manual.

10. Artefatos de build versionados simultaneamente
- dist atual e tarball antigo com hashes diferentes aumentam superficie de confusao operacional.

Backend

  - impacto se incorreto: fallback quebra e sync gera credencial invalida

- ORTHANC_CREDENTIALS_PATH
  - default: /etc/orthanc/credentials.json
  - uso: escrita do arquivo no sync
  - impacto se incorreto/sem permissao: sync falha, credenciais desatualizadas

- JWT_SECRET
  - default: 332291180@*p
  - uso: assinatura JWT + chave Fernet derivada
  - impacto se divergente/rotacionado: token invalido e/ou decrypt de password_encrypted falha

- JWT_EXPIRATION_HOURS
  - valor fixo: 12
  - impacto: expira sessao de auth nesse intervalo

- DB_PATH
  - default: /opt/healthlin/backend/healthlin.db
  - uso: conexao SQLite
  - impacto se incorreto/sem escrita: autenticao e gestao de usuarios indisponiveis

  - default: ../viewer/dist absoluto
  - service: /opt/healthlin/frontend/dist
  - uso: fallback SPA no Flask
  - impacto se incorreto: API sobe, frontend nao e servido (404 no catch-all)

- FLASK_ENV
  - uso: gate de seguranca para JWT_SECRET em producao

Frontend
- VITE_API_URL
  - default: ''
  - uso 1: axios baseURL
  - uso 2: URL wadouri do arquivo DICOM
  - impacto se incorreto: chamadas API e/ou carregamento DICOM falham
